A hands-on, runnable walkthrough using video — with real, executed output.

Long video understanding hit an inflection point in 2026. Linear-scaling Video VLMs — models that grow their context proportionally rather than quadratically with frame count — finally made hour-long video inference practical without melting your GPU budget. Between Veo pushing generation quality forward and ffmpeg remaining the universal knife for frame extraction, the tooling side of video AI is maturing fast. But the infrastructure between "I have frames" and "I have answers" remains a mess of bespoke scripts, and that gap grows wider as fine-tuning and inference pipelines get more complex.

The core problem isn't the model. It's everything around it: extracting frames, resizing them, routing them through embeddings or VLMs, storing results alongside the source material, and keeping all of it consistent when you add new data. Most teams end up with a tangle of shell scripts piping ffmpeg output into Python glue that writes to three different stores. It works until it doesn't.

Pixeltable takes a different approach. It gives you a declarative table where transforms are computed columns — define them once, and they stay in sync as data arrives. No DAGs, no orchestrators, no batch jobs to babysit. The same pattern works whether your column holds a scalar, an image, a video clip, or a model output.

Computed columns in practice

The concept is easier to show than explain. Here we build a multimodal image table, then add computed columns that derive new values automatically. Every future insert gets the same transforms with zero additional code.

import pixeltable as pxt
from PIL import Image

# A fresh, isolated workspace.
pxt.drop_dir("demo", force=True)
pxt.create_dir("demo")

# Generate a few local images so the example is fully offline.
import tempfile, os
tmp = tempfile.mkdtemp()
paths = []
for i, color in enumerate([(220, 50, 50), (50, 180, 90), (60, 90, 220)]):
    p = os.path.join(tmp, f"img_{i}.png")
    Image.new("RGB", (256 + i * 32, 192), color).save(p)
    paths.append(p)

# Declare a table with an image column and insert the rows.
t = pxt.create_table("demo.images", {"img": pxt.Image, "label": pxt.String})
t.insert([{"img": p, "label": l} for p, l in zip(paths, ["red", "green", "blue"])])

# Computed columns: derived automatically and kept in sync on insert.
t.add_computed_column(width=t.img.width)
t.add_computed_column(height=t.img.height)
t.add_computed_column(thumb=t.img.resize([96, 96]))

print(t.select(t.label, t.width, t.height).collect())

Here is the actual output from running that snippet:

label  width  height
0    red    256     192
1  green    288     192
2   blue    320     192

That's real execution, not a mockup. The width, height, and thumb columns were derived automatically at insert time. Add a thousand more rows tomorrow and every one gets the same transforms without touching the pipeline.

Why this matters for video workflows

Now picture the video version. Replace the image column with a video column. Replace t.img.width with a computed column that calls ffmpeg to extract keyframes, another that runs each frame through a VLM for captioning, another that generates embeddings for retrieval. The shape of the code stays the same — declare the table, define the transforms, insert data. Pixeltable handles execution order, caching, and incremental updates.

This is the piece that linear-scaling VLMs need on the infrastructure side. A model that can process an hour of video in linear time is only useful if the pipeline feeding it frames, metadata, and context can keep up without collapsing into a maintenance burden. Declarative computed columns solve that: you describe what you want derived, and the system figures out how and when to compute it.

Try it yourself

pip install pixeltable
