A hands-on, runnable walkthrough using multimodal — with real, executed output.

Expanse's YC launch put a spotlight on something most teams already feel: GPU capacity is not the bottleneck anymore. The bottleneck is the glue code between your data and the models that consume it. Every multimodal pipeline -- whether it's embedding video frames for RAG, routing images through an LLM agent, or indexing audio for search -- follows the same pattern: load data, transform it, derive new columns, keep everything in sync. And every team writes that plumbing from scratch, poorly, in a tangle of scripts that nobody wants to maintain.

Pixeltable replaces that tangle with a single abstraction: a declarative table where transforms are computed columns. You define the derivation once. New rows get the same transforms automatically. No DAG orchestrator, no batch scheduler, no glue.

Build a multimodal image table in 20 lines

The best way to see what this means is to run it. The example below creates a table of images, then adds computed columns that extract width, height, and a thumbnail -- all derived automatically from the source column. No loops, no manual bookkeeping.

import pixeltable as pxt
from PIL import Image

# A fresh, isolated workspace.
pxt.drop_dir("demo", force=True)
pxt.create_dir("demo")

# Generate a few local images so the example is fully offline.
import tempfile, os
tmp = tempfile.mkdtemp()
paths = []
for i, color in enumerate([(220, 50, 50), (50, 180, 90), (60, 90, 220)]):
    p = os.path.join(tmp, f"img_{i}.png")
    Image.new("RGB", (256 + i * 32, 192), color).save(p)
    paths.append(p)

# Declare a table with an image column and insert the rows.
t = pxt.create_table("demo.images", {"img": pxt.Image, "label": pxt.String})
t.insert([{"img": p, "label": l} for p, l in zip(paths, ["red", "green", "blue"])])

# Computed columns: derived automatically and kept in sync on insert.
t.add_computed_column(width=t.img.width)
t.add_computed_column(height=t.img.height)
t.add_computed_column(thumb=t.img.resize([96, 96]))

print(t.select(t.label, t.width, t.height).collect())

Here is the actual output from running that snippet:

label  width  height
0    red    256     192
1  green    288     192
2   blue    320     192

That is not a mockup. The code executed, the table was created, and those are the real values. The interesting part is what happens next: insert a fourth row and Pixeltable computes width, height, and thumb for it without you asking. The column definitions are the pipeline.

Why this matters for the GPU-capacity conversation

Expanse and projects like it are making raw compute cheaper and more available. That shifts the constraint. When GPU hours are abundant, the limiting factor becomes how fast you can get data into and out of models -- and how reliably your transforms stay consistent as data changes. This is exactly the problem computed columns solve.

Swap the toy transforms above for real ones -- a CLIP embedding column, a frame-extraction UDF over video, an LLM classification call -- and the shape of the code stays identical. You declare the column, Pixeltable handles execution, caching, and incremental updates. The same table can hold images, video, audio, and text alongside scalar metadata. Multimodal is not a special mode; it is the default.

Where to go from here

Install and run the example above in under a minute:

pip install pixeltable

From there, the practical next steps: add an embedding column backed by a real model, attach a similarity index, and query it. The docs walk through each of those. The underlying point stays the same -- stop writing orchestration code and start declaring what you want computed.
